# 工业互联网平台总体架构设计

## 1. 文档目标

本文档基于现有项目规划，输出系统首版总体架构方案，用于指导后续原型设计、后端开发、数据库建模和接口定义。

本次设计遵循以下原则：

1. 前端以可演示为目标，不过度投入 UI 细节，以快速串联业务闭环为主。
2. 后端围绕主数据、工程建模、成本核算、LCC 仿真、IoT 数据接入五类核心能力展开。
3. 数据库设计优先服务领域模型和计算链路，保证后续算法扩展、版本管理和审计追溯。
4. 架构优先采用单体分层 + 模块化领域设计，降低项目初期复杂度，同时保留后续服务化拆分空间。

---

## 2. 项目定位与业务主线

系统面向工业互联网场景，核心目标是围绕产品设计方案、工艺路线、资源能力和现场运行数据，构建一套可支持成本建模、生命周期成本仿真和优化分析的平台。

核心业务主线如下：

1. 建立主数据底座：材料、设备、人员、工艺、能源、单位换算、仿真目标参数。
2. 完成工程建模：将产品结构、设计方案、物料、工艺、设备、人员形成可计算模型。
3. 执行成本核算：完成静态成本估算、作业成本映射和产品成本明细生成。
4. 执行 LCC 仿真：引入设备能力、健康度、能源价格和人员约束进行多方案模拟。
5. 形成分析与优化：输出多方案对比、优化建议和决策驾驶舱结果。

因此，系统的技术重心不在页面视觉，而在于：

1. 数据模型是否能承载复杂工业对象。
2. 计算链路是否可追溯、可重算、可版本化。
3. 仿真任务是否支持异步运行、过程记录和结果归档。

## 3. 总体技术选型

### 3.1 前端技术栈

前端定位为业务演示型应用，建议采用：

1. Vue 3
2. Vite
3. TypeScript
4. Vue Router
5. Pinia
6. Element Plus
7. ECharts

前端职责聚焦在：

1. 表单录入与台账管理
2. 产品结构树和工艺编排界面
3. 成本台账与仿真结果展示
4. 数字孪生看板的二维演示版

前端不承担复杂业务逻辑，所有规则校验、成本计算、仿真编排与结果解释均放在后端。

### 3.2 后端技术栈

后端建议采用：

1. FastAPI：提供 REST API 和任务接口
2. SQLAlchemy 2.0：ORM 和复杂查询构建
3. Pydantic：接口模型和参数校验
4. Alembic：数据库版本管理
5. MySQL 8.0：核心业务数据存储
6. Redis：缓存、任务状态、队列中间件
7. Celery：异步仿真任务和批量计算任务调度
8. MQTT/OPC-UA 适配层：IoT 数据采集接入

### 3.3 基础设施建议

1. Nginx：前端静态资源托管与反向代理
2. Docker Compose：开发和演示环境部署
3. 对象存储或本地文件存储：方案附件、导入文件、报表文件
4. 日志采集：按应用日志、审计日志、任务日志分层存储

## 4. 总体架构方案

## 4.1 架构风格

建议首期采用模块化单体架构。

原因如下：

1. 当前项目处于业务建模和核心算法验证阶段，领域规则尚未稳定。
2. 直接拆成微服务会显著增加接口编排、事务一致性和运维复杂度。
3. 成本核算和仿真过程高度依赖主数据与工程模型，模块间耦合强，先在单体内部做清晰边界更合适。

建议形态：

1. 一个 Vue3 Web 前端
2. 一个 FastAPI 主应用
3. 一个 Celery Worker
4. 一个 MySQL 实例
5. 一个 Redis 实例
6. 一个 IoT 接入适配组件

后续若业务增长，可逐步拆分为主数据服务、建模服务、仿真服务、IoT 接入服务。

## 4.2 逻辑分层

后端建议采用四层结构：

1. 接口层 API Layer
   - 提供 REST API
   - 处理认证、参数校验、分页、文件上传下载

2. 应用层 Application Layer
   - 编排业务用例
   - 控制事务边界
   - 调用领域服务、仓储、任务调度器

3. 领域层 Domain Layer
   - 承载核心领域对象和规则
   - 包括成本规则、单位换算、工艺资源挂载、LCC 仿真逻辑

4. 基础设施层 Infrastructure Layer
   - MySQL 仓储实现
   - Redis 缓存
   - Celery 任务
   - IoT 采集适配
   - 文件存储与日志

## 4.3 系统上下游关系

```text
Vue3 Demo Frontend
    |
    v
FastAPI API Gateway / Application
    |---------------- 主数据域
    |---------------- 工程建模域
    |---------------- 成本核算域
    |---------------- 仿真优化域
    |---------------- IoT 接入域
    |---------------- 系统管理域
    |
    |---- MySQL
    |---- Redis
    |---- Celery Worker
    |---- MQTT/OPC-UA Adapter
```

## 5. 领域模块划分

结合当前规划，建议后端按以下 6 个领域模块组织。

### 5.1 全景视界域

职责：

1. 聚合车间实时状态、设备状态、产线指标、成本指标。
2. 面向首页驾驶舱和数字孪生看板输出聚合接口。

特点：

1. 以读模型为主。
2. 数据来源于 IoT 实时数据、成本台账、仿真结果快照。

### 5.2 主数据与资源中心域

职责：

1. 管理材料、设备、人员、工艺、能源、单位换算、KPI 参数。
2. 支撑柔性属性建模和资源挂载。
3. 为成本与仿真提供标准化数据底座。

这是系统最核心的底层域之一。

### 5.3 工程建模域

职责：

1. 管理产品、项目、设计方案、版本。
2. 维护产品结构树、零部件、材料匹配、工艺路线、设备指派。
3. 将设计对象映射为可计算模型。

这是连接“设计数据”和“成本仿真数据”的桥梁域。

### 5.4 成本核算域

职责：

1. 管理作业成本规则。
2. 根据工程模型生成静态成本台账。
3. 输出产品级、部件级、工序级成本明细。

### 5.5 仿真优化域

职责：

1. 发起仿真任务。
2. 拉取方案版本、资源约束、设备实时状态、能源价格。
3. 执行多方案 LCC 模拟。
4. 归档仿真过程、结果、对比分析和优化建议。

该模块建议采用异步任务机制实现。

### 5.6 IoT 与平台管理域

职责：

1. 接入设备测点数据。
2. 管理实时数据策略和历史归档。
3. 管理组织、用户、角色、权限、日志和审计。

## 6. 关键业务流设计

## 6.1 主数据建模流

1. 管理员定义属性字典、单位、资源分类和模板。
2. 业务人员维护材料、设备、人员、能源费率等实例数据。
3. 系统完成变量标识、量纲校验、换算闭包和版本生效管理。

## 6.2 工程方案建模流

1. 创建项目和产品。
2. 创建设计方案及版本。
3. 导入或录入产品结构树。
4. 为零部件匹配材料。
5. 为零部件配置工艺路线。
6. 为工艺节点挂载设备、人员、辅材和工时。
7. 固化为可计算的方案快照。

## 6.3 静态成本核算流

1. 选择设计方案版本。
2. 调用成本规则引擎。
3. 逐工序解析设备折旧、能耗、人工、材料损耗。
4. 汇总输出产品成本台账和成本明细。

## 6.4 LCC 仿真流

1. 用户选择多个设计方案。
2. 配置仿真目标、时间范围、约束条件和场景参数。
3. 系统创建仿真任务并生成输入快照。
4. Celery Worker 异步执行仿真。
5. 仿真过程按阶段记录日志和中间结果。
6. 任务完成后写入结果表和分析结论。
7. 前端轮询或订阅任务状态，并展示结果。

## 7.1 推荐目录结构

说明：

5. services 负责编排用例。
6. domain 放领域规则和计算逻辑。
7. tasks 放 Celery 异步任务。

## 7.2 核心服务拆分

建议至少包括以下应用服务：

1. 字典与模板服务
2. 材料主数据服务
3. 设备能力服务
4. 人员资质服务
5. 工艺资源包服务
6. 产品方案服务
7. 产品结构建模服务
8. 成本计算服务
9. 仿真任务服务
10. 仿真结果分析服务
11. IoT 接入服务
12. 认证授权服务

## 7.3 异步任务设计

建议以下场景使用 Celery：

1. LCC 仿真任务
2. 成本批量重算
3. 产品结构批量导入
4. IoT 历史数据归档和清洗
5. 报表导出

任务状态建议统一为：

1. PENDING
2. RUNNING
3. SUCCESS
4. FAILED
5. CANCELED

每个任务都需要保存：

1. 发起人
2. 输入参数快照
3. 进度百分比
4. 执行日志
5. 结果摘要
6. 异常信息

## 8. MySQL 数据库总体设计

## 8.1 设计原则

1. 核心业务数据使用关系模型，保证一致性和可追溯性。
2. 柔性属性采用 JSON 字段承载，但关键查询字段必须冗余到结构化列。
3. 所有关键业务对象引入版本号、状态、创建人、更新时间、逻辑删除标记。
4. 仿真和成本计算必须保留快照表，避免基础数据变化导致历史结果不可重现。

## 8.2 数据域划分

建议数据库表按以下主题划分：

1. 系统管理域表
2. 主数据域表
3. 工程建模域表
4. 成本核算域表
5. 仿真优化域表
6. IoT 数据域表
7. 日志审计域表

## 8.3 核心表设计建议

### A. 系统管理域

1. org_department
2. sys_user
3. sys_role
4. sys_permission
5. sys_user_role
6. sys_role_permission
7. sys_audit_log

### B. 主数据域

1. md_attr_definition：属性定义表，存储属性名称、变量编码、数据类型、单位、适用资源类型。

1. md_unit_dimension：量纲定义表，如长度、质量、体积、时间、能耗。

1. md_unit：单位表，如 mm、m、kg、t、h、kwh。

1. md_unit_conversion：同量纲标准换算表。

1. md_resource_category：资源分类树，支撑材料、设备、人员、工具分类。

1. md_material：材料主表，包含编码、名称、计价单位、损耗率、回收残值率、JSON 属性。

1. md_material_conversion：材料专属跨维度换算表，例如长度到质量的专属换算规则。

1. md_material_substitute：材料替代关系表。

1. md_equipment：设备主表，包含设备编码、IoT 标识、折旧费率、能耗系数、能力边界、JSON 属性。

1. md_labor_skill：工种技能定义表。

1. md_labor_rate：工种等级费率表。

1. md_process_template：标准工艺模板表。

1. md_process_step：标准工艺步骤表。

1. md_process_step_resource：工艺步骤资源挂载表，关联设备、人工、辅材、工时配置。

1. md_energy_calendar：能源价格日历表，支持峰平谷、日期范围、生效区间。

1. md_simulation_kpi：仿真指标字典表。

### C. 工程建模域

1. eng_project
2. eng_product
3. eng_product_version
4. eng_design_scheme
5. eng_design_scheme_version
6. eng_bom_node
7. eng_component_material_match
8. eng_component_process_route
9. eng_route_step_bind
10. eng_model_snapshot

说明：

1. 产品、方案、版本要分离，避免直接覆盖。
2. eng_bom_node 用树结构存储产品结构，可采用 parent_id 方式。
3. eng_model_snapshot 用于固化某一版本方案的完整可计算输入。

### D. 成本核算域

1. cost_rule_set
2. cost_rule_item
3. cost_calc_task
4. cost_product_ledger
5. cost_component_detail
6. cost_process_detail

说明：

1. cost_product_ledger 保存一次核算的产品级结果。
2. cost_component_detail 保存部件级拆解成本。
3. cost_process_detail 保存工序级明细，是后续分析和审计重点。
4. cost_factor_snapshot 保存核算所使用的资源费率快照。

### E. 仿真优化域

1. sim_scenario
2. sim_task
3. sim_task_scheme
4. sim_constraint
5. sim_input_snapshot
6. sim_stage_result
7. sim_scheme_result
8. sim_optimization_advice

说明：

1. sim_task 是仿真主任务表。
2. sim_task_scheme 维护任务与方案的多对多关系。
3. sim_input_snapshot 保存任务运行时的完整输入快照。
4. sim_stage_result 记录生命周期分阶段结果。
5. sim_scheme_result 保存单个方案汇总结果。

## 9. 核心数据设计要点

## 9.1 柔性属性设计

材料、设备等对象不可完全依赖固定字段，建议采用“结构化字段 + JSON 扩展字段”模式：

1. 固定字段承载高频过滤条件和关键指标。
2. JSON 字段承载低频变化属性。
3. 属性字典表负责解释 JSON 中每个变量的业务含义。

这样既保证扩展性，也避免全表无限加字段。

## 9.2 快照设计

仿真和成本计算必须做快照，至少包括：

1. 方案快照
2. 工艺资源包快照
3. 费率快照
4. 能源价格快照
5. 设备状态快照

原因：

1. 防止主数据变更后历史结果失真。
2. 支撑审计追溯。
3. 支撑结果复算和差异分析。

## 9.3 版本设计

以下对象必须支持版本管理：

1. 产品方案
2. 工艺模板
3. 成本规则集
4. 仿真场景模板
5. 材料和设备关键费率参数

建议采用“主表 + 版本表 + 发布状态”设计，而不是直接覆盖更新。

6. /api/v1/engineering/schemes
7. /api/v1/costing/tasks
8. /api/v1/simulations/tasks
9. /api/v1/dashboard
10. /api/v1/iot
11. /api/v1/system

接口设计要求：

1. 列表接口支持分页、筛选、排序。
2. 关键对象支持版本查询和快照查询。
3. 长耗时任务采用“创建任务 + 查询状态 + 查询结果”模式。
4. 文件导入导出采用独立接口。

## 13. 非功能性要求

### 13.1 安全性

1. JWT 或 Session + Token 认证
2. 基于 RBAC 的权限控制
3. 核心数据按组织、项目或方案维度隔离
4. 全量保留审计日志

### 13.2 性能

1. 高频查询页面优先做聚合读模型
2. 仿真任务异步化，避免阻塞 Web 请求
3. 热点字典和配置可缓存在 Redis

### 13.3 可维护性

1. 统一异常码和错误响应模型
2. Alembic 管理数据库变更
3. 单元测试优先覆盖领域规则和成本计算逻辑
4. 核心计算过程保留日志和中间结果

## 15. 结论

本项目首版最合适的技术路线是：

1. 前端采用 Vue3，聚焦业务演示，不追求复杂交互和精细化视觉。
2. 后端采用 FastAPI + 模块化单体架构，优先沉淀领域规则、成本计算和仿真任务能力。
3. 数据库采用 MySQL，围绕主数据、工程模型、成本明细、仿真快照建立可扩展关系模型。
4. 异步任务采用 Celery + Redis，承载仿真和批量计算。
5. 系统建设顺序应优先保证“主数据可建模、方案可计算、结果可追溯”，再逐步增强 IoT 接入和数字孪生展示。

该架构能够满足当前 Demo 阶段的落地要求，同时为后续扩展成正式工业平台预留足够空间。
